4 - Control de versiones y documentación

Documentación

“La documentación que se elabora para una aplicación, a pesar de ser un documento técnico que ayude al futuro desarrollo de ésta, debe contener una parte en un lenguaje sencillo para ayudar al usuario final a entender el funcionamiento de la aplicación.”

Una buena práctica a la hora de desarrollar un software consiste en preparar una documentación detallada sobre las especificaciones del programa. En muchas ocasiones, el personal técnico omite esta acción, ya sea por falta de conocimientos, motivación o presupuesto, o porque no suele considerarse una necesidad básica del proceso de desarrollo. A pesar de ello, es importante señalar que se trata de una práctica obligatoria para todo desarrollador de aplicaciones web.

Documentación | Importancia de la documentación

¿Por qué es importante documentar?

“Es importante señalar que se trata de una práctica obligatoria para todo desarrollador de aplicaciones web.”
  • Facilita el mantenimiento: permite entender cómo funciona una aplicación, qué tecnologías utiliza y cómo está estructurada. Esto es clave cuando se deben corregir errores o añadir nuevas funcionalidades.
  • Mejora la colaboración entre desarrolladores: asegura que todos comprendan el propósito y funcionamiento del código, evitando malentendidos y duplicaciones de esfuerzo.
  • Permite escalar o evolucionar el proyecto con seguridad: facilita la adaptación a nuevas necesidades sin comprometer la integridad del proyecto.
  • Ayuda a integrar nuevas personas al equipo: nuevos programadores pueden ponerse al día rápidamente con una guía técnica clara.
  • Evita la dependencia del autor original del código: sin documentación, solo quien escribió el código lo entiende. Esto es riesgoso si esa persona no está disponible.
  • Es obligatoria en entornos profesionales y formativos.

Ejemplo real:

En un equipo de tres personas desarrollando con Node.js, si cada uno usa diferentes convenciones o no comenta el código, los errores y malentendidos son inevitables.

Imagina que heredas una aplicación REST hecha en PHP que consulta una base de datos. Sin documentación, no sabes:

  • Qué endpoints existen.
  • Qué parámetros esperan.
  • Qué hacen exactamente.

Tipos y formatos de documentación:

  • Técnica: estructura, lenguajes, librerías, configuración.
  • Funcional: cómo usar la aplicación.
  • De instalación/despliegue.
  • De API: especialmente para REST.

Documentación | Tipos y formatos

Formatos frecuentes:

  • Texto (Word, PDF).
  • Formato web interactivo (Markdown + HTML).
  • Auto-generada desde el código fuente.

1. Markdown (.md)

  • Sencillo, rápido de escribir, legible y soportado por todas las plataformas de control de versiones.
  • Ideal para comenzar un proyecto o mantener documentación viva.
  • Limitado para documentación técnica compleja (API, tipos de datos, esquemas…).

2. YAML o JSON (OpenAPI 3)

  • Estándar en el mundo REST para describir APIs.
  • Compatible con herramientas como Swagger, Redoc, Stoplight.
  • Requiere cierta estructura y conocimiento técnico.

3. HTML

  • Permite documentación rica, navegable e interactiva.
  • Usado con generadores como Docusaurus, MkDocs, GitBook.
  • Requiere hosting y a menudo herramientas extra.

4. PDF

  • Perfecto para entregar documentación estática, formal o académica.
  • Aceptado como estándar para documentación legal o para clientes.
  • No editable fácilmente ni versionable.

5. XML

  • Todavía se usa en sistemas legacy, configuración (Maven, Android), y estandarización (WSDL, SOAP).
  • Ha sido reemplazado por JSON y YAML en la mayoría de casos modernos.

Documentación | Control de versiones

¿Qué es?

“Seguimiento de las versiones de cada aplicación, porque a lo largo de la vida de una aplicación deberán realizarse cambios.”

Funciones principales:

  • Historial de cambios.
  • Volver a versiones estables.
  • Colaboración simultánea.
  • Prevención de conflictos.

Ejemplos de comandos básicos de Git:

  • git init
  • git add
  • git commit -m "Primera versión de la API de productos"

Documentación | Herramientas de control de versiones

  • Git (CLI o con interfaz como GitHub Desktop)
  • GitHub, GitLab, Bitbucket

Operaciones comunes:

  • clone, commit, push, pull, merge, branch

Documentación | Buenas prácticas

“Es importante documentar también todos estos cambios para que la información no se quede obsoleta o existan incompatibilidades entre versiones.”

Recomendaciones clave:

  • Comenzar la documentación desde el primer día.
  • Documentar cada endpoint REST.
  • Añadir comentarios relevantes en PHP/JS.
  • Usar plantillas y estilos homogéneos.
  • Actualizar siempre junto al código.
